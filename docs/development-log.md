### Ключевые архитектурные решения

**Микросервисная архитектура из трех контейнеров.**
- `web` (Streamlit) — пользовательский интерфейс, порт 8501
- `api` (FastAPI) — основная бизнес-логика и оценка надежности, порт 8000
- `ml` (FastAPI + Markov) — генерация паролей, доступен только внутри Docker-сети

Обоснование: соответствие принципу единой ответственности (SRP); изоляция зависимостей; снижение поверхности атаки за счет изоляции ML-сервиса от внешней сети; независимое тестирование и обновление компонентов.

**Выбор технологического стека.**
- Python 3.11 — основной язык, выбран из-за богатой экосистемы NLP/ML и простоты прототипирования
- FastAPI — современный веб-фреймворк для построения REST API, генерирует автоматическую OpenAPI-документацию
- Streamlit — минимизирует время разработки UI, позволяет собрать веб-интерфейс на чистом Python
- Docker Compose — стандарт оркестрации многоконтейнерных приложений

**Выбор Марковской модели вместо более сложных подходов** (LSTM, GAN, LLM).

`[ВКР]` Классические методы оценки надежности паролей (вычисление энтропии Шеннона, проверка по словарям) хорошо решают задачу оценки, но плохо — задачу генерации **правдоподобных** паролей: их вывод либо предсказуем, либо несжимаем для запоминания человеком. Применение методов искусственного интеллекта оправдано именно в задаче генерации.

Среди ИИ-моделей для моделирования паролей рассматривались:
- Марковские модели (символьные n-граммные)
- Рекуррентные нейронные сети (LSTM)
- Генеративно-состязательные сети (PassGAN)
- Большие языковые модели (LLM)

Выбрана **Марковская модель** по следующим основаниям:
1. **Промышленный стандарт.** Используется в Hashcat (`.hcstat`-файлы) и John the Ripper — двух главных инструментах взлома паролей. Является базовой моделью угрозы, против которой проверяется надежность паролей.
2. **Прозрачность.** Каждое решение модели объяснимо через вероятностную таблицу переходов. Соответствует принципам explainable AI.
3. **Эффективность.** Не требует GPU, обучается за минуты, работает на любом CPU.
4. **Академическая база.** Применение к моделированию паролей описано в работах Castelluccia (2012), Narayanan, Shmatikov (2005).

### Цитируемые формулировки для ВКР

`[ВКР]` Архитектура приложения построена на принципе разделения ответственности: пользовательский интерфейс, бизнес-логика и модель машинного обучения вынесены в независимые контейнеры, взаимодействующие по HTTP-протоколу.

`[ВКР]` Контейнеры с моделью машинного обучения и хранилищем обучающих данных не экспонированы во внешнюю сеть и доступны только из внутренней Docker-сети. Это снижает поверхность атаки: даже при компрометации фронтенд-сервиса злоумышленник не получает прямого доступа к модели и данным.

`[ВКР]` В Dockerfile применена стандартная оптимизация многослойного кэширования: файл `requirements.txt` копируется и устанавливается отдельным шагом, до копирования основного кода. Это позволяет переиспользовать слой с установленными зависимостями при изменениях исходного кода, что существенно ускоряет цикл итеративной разработки.

`[ВКР]` Для предотвращения арифметического исчезновения порядка (numerical underflow) при перемножении большого числа малых вероятностей оценка пароля вычисляется в логарифмическом пространстве как сумма логарифмов условных вероятностей.

`[ВКР]` Применение сглаживания Лапласа (add-one smoothing) обеспечивает корректное вычисление оценки для паролей, содержащих биграммы, не встречавшиеся в обучающей выборке, исключая обнуление совокупной вероятности.

|----------|-----------|-----------|
| Уникальных префиксов | 99 | 118 |
| Копий обучающего набора (из 20 сгенерированных) | 8 (40%) | 20 (100%) |
| Уникальных сгенерированных | 18 (90%) | 13 (65%) |

При order=2 модель генерирует гибриды (`prinistno1`, `pasebassword1`, `letmeine`), комбинируя фрагменты разных обучающих паролей через общие биграммы. При order=3 на том же корпусе модель полностью воспроизводит обучающую выборку — типичный случай переобучения, обусловленный недостаточным количеством обучающих данных относительно сложности модели.

`[ВКР]` Экспериментально показан компромисс между сложностью модели и объемом обучающей выборки: на корпусе из 23 паролей модель порядка 3 полностью переобучается и теряет способность к обобщению, в то время как модель порядка 2 сохраняет способность генерировать новые правдоподобные комбинации. На полном корпусе RockYou (10⁷ паролей) ожидается обратное соотношение — модель более высокого порядка точнее моделирует распределение.

### Ключевые архитектурные решения

**Применение паттерна `lifespan` для загрузки модели.**

Альтернативой было бы либо обучать модель **при каждом запросе** (катастрофически медленно), либо загружать ее на уровне модуля при импорте (привязка к деталям импорта, сложнее тестировать). `lifespan` — стандартный для современного FastAPI способ выполнить инициализацию **один раз** при запуске сервиса. Модель обучается в момент старта контейнера и затем хранится в оперативной памяти процесса; каждый последующий запрос пользуется уже готовым объектом.

`[ВКР]` Обучение модели выполняется однократно на этапе запуска сервиса с использованием паттерна `lifespan` фреймворка FastAPI. Это обеспечивает константное время отклика на запросы пользователей вне зависимости от объема обучающей выборки: стоимость обучения амортизируется на все последующие запросы за время жизни процесса.

**Использование Pydantic для валидации входов и выходов.**

Pydantic-модели описывают **структуру** ожидаемого JSON и его **ограничения**: типы полей, минимальные и максимальные длины, диапазоны значений. FastAPI применяет эти описания автоматически при разборе входящих запросов — невалидные данные отклоняются до того, как попадут в бизнес-логику. На основе тех же моделей генерируется OpenAPI-документация (Swagger UI), что устраняет рассинхронизацию между кодом и описанием API.

`[ВКР]` Для контроля корректности входных данных применен механизм декларативной валидации на базе библиотеки Pydantic. Структура JSON-запросов и ответов описывается типизированными классами, ограничивающими допустимые значения (длины строк, диапазоны чисел). Валидация выполняется автоматически перед передачей данных в обработчик; при нарушении ограничений сервис возвращает структурированную ошибку с кодом HTTP 422 (Unprocessable Entity), не достигая бизнес-логики.

**Автогенерация документации API (OpenAPI/Swagger).**

FastAPI по умолчанию строит документацию API на основе кода — из сигнатур функций-обработчиков, Pydantic-моделей и docstring. Документация доступна в интерактивной форме (Swagger UI) по пути `/docs` и в машинно-читаемом виде (OpenAPI 3.1 JSON) по `/openapi.json`. Это закрывает требование к описанию интерфейсов без дополнительной работы и обеспечивает синхронизацию документации и реализации.

`[ВКР]` Документация программного интерфейса генерируется автоматически на основе исходного кода и аннотаций типов в соответствии со спецификацией OpenAPI 3.1. Интерактивная форма документации (Swagger UI) доступна по пути `/docs` и позволяет тестировать эндпоинты непосредственно из браузера, что использовалось при разработке для верификации поведения сервиса.

**Логирование запросов на уровне ASGI-сервера.**

Uvicorn ведет журнал каждого HTTP-запроса с указанием источника, метода, пути и кода ответа. Этот журнал доступен через стандартные средства Docker (`docker compose logs`). Для базовой наблюдаемости сервиса дополнительная инструментация не требуется.

### Ключевые архитектурные решения

**Локальная база утечек вместо обращения к HIBP API.**

Альтернативный подход — использовать публичный API сервиса Have I Been Pwned. Это бы давало всегда актуальную базу и не требовало хранения данных у себя. Однако был выбран офлайн-вариант со списком топ-100k паролей по следующим основаниям:

1. **Приватность пароля пользователя.** При обращении к внешнему API, даже с механизмом k-anonymity, фрагмент хеша пароля передается за пределы системы. В офлайн-варианте пароль не покидает границ нашего сервиса.
2. **Производительность.** Запрос к внешнему API — это 100–500 мс сетевой задержки. Локальная проверка по `set` хешей — наносекунды (O(1)).
3. **Автономность.** Система работает без подключения к интернету, что важно для развертывания в изолированных корпоративных сетях.
4. **Покрытие угроз.** Распределение паролей в утечках сильно скошено: топ-1000 паролей покрывают огромную долю реальных компрометаций, топ-100k — практически все случаи использования общеизвестных паролей.

`[ВКР]` Проверка пароля по базе известных утечек реализована в офлайн-режиме на основе локальной копии списка наиболее распространенных паролей из публичного репозитория SecLists. Это устраняет необходимость передачи каких-либо производных пароля за пределы доверенной границы системы и обеспечивает работу проверки без зависимости от внешних сетевых сервисов.

**Хеширование паролей при загрузке базы утечек.**

При загрузке `top100k.txt` каждый пароль немедленно преобразуется в SHA-1 хеш, исходные строки в памяти не сохраняются. Это снижает риск компрометации данных при возможном дампе памяти процесса api: даже при получении дампа атакующий увидит только хеши.

`[ВКР]` Список паролей из базы утечек хранится в памяти сервиса в хешированном виде (алгоритм SHA-1, совместимый с API сервиса Have I Been Pwned). Исходные пароли немедленно отбрасываются после хеширования, что снижает риск их компрометации при возможной утечке состояния процесса.

**Асинхронный HTTP-клиент для межсервисной коммуникации.**

Выбран httpx (асинхронный клиент) вместо более распространенного requests (синхронного). Это согласуется с асинхронной природой FastAPI: пока api ожидает ответ от ml, исполнитель может обрабатывать другие входящие запросы, не блокируя поток. Для одиночного клиента эффект незначителен, но при масштабировании дает существенный прирост пропускной способности.

**Итоговый вердикт с приоритетом сигнала об утечке.**

При формировании итоговой оценки пароля проверка по утечкам имеет наивысший приоритет: если пароль найден в базе утечек, итоговый вердикт — «КРИТИЧЕСКИ СЛАБЫЙ» независимо от оценок zxcvbn и Марковской модели. Это отражает тот факт, что наличие пароля в публичной утечке является **прямым свидетельством** его компрометации, в то время как остальные оценки являются лишь статистическими прогнозами вероятности подбора.

`[ВКР]` Итоговый вердикт о надежности пароля строится с приоритетом проверки по базе утечек: совпадение с известным утекшим паролем переопределяет любые статистические или эвристические оценки. Данное проектное решение отражает фундаментальное различие между предсказанием вероятности компрометации (что делают zxcvbn и Марковская модель) и прямым доказательством компрометации (что дает база утечек).

### Замечания

- База утечек загрузилась как 99 839 записей вместо 100 000. Расхождение объясняется наличием дубликатов или пустых строк в исходном файле top100k.txt — обычное явление в публичных дампах. Для целей проверки это не критично.
- Пример с паролем `Tr0ub4dor&3` (известный из комикса XKCD «Password Strength») иллюстрирует ограничение эвристического подхода: пароль, построенный по предсказуемому шаблону, оценен zxcvbn как сильный (4/4). Это является основанием для использования **множественных** методов оценки, что и реализовано в нашей системе.

### Решенные технические проблемы

**Использование одного volume `data/` в двух сервисах.**

И ml, и api нуждаются в доступе к файлам из `data/` (rockyou.txt для обучения модели, top100k.txt для базы утечек). Поскольку оба монтирования read-only, это не создает конфликтов чтения/записи. Альтернативой было бы разделение данных на два отдельных volume, но это бы усложнило структуру без существенной выгоды: оба файла принадлежат к одной категории данных (списки паролей), оба читаются только при старте сервисов.
